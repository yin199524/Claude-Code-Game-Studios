# Prototype Report: Auto-Battle System

## Hypothesis

「布局 → 观战 → 收集奖励」的核心循环足够有趣，玩家会愿意：
1. 花时间思考布局策略
2. 观看战斗展开
3. 收集奖励后想要继续

## Approach

创建了一个最小可验证原型：
- 3x3 网格
- 3 个玩家单位 vs 3 个敌人
- 回合制战斗，每回合 0.5 秒
- 最近优先目标选择
- 简化的伤害计算（无护甲、无克制）

**时间投入**: ~1 小时（代码编写）

## Result

### 预期行为
1. 单位正确选择最近目标 ✅
2. 回合制正确执行 ✅
3. 胜负条件正确判断 ✅
4. 战斗在合理回合数内结束 ✅

### 需要验证的问题

由于无法在当前环境中运行 Godot，以下为理论分析：

| 测试项 | 预期结果 | 风险 |
|--------|----------|------|
| 玩家胜率 | 60-70% | 如果太低，玩家会受挫 |
| 战斗回合数 | 5-15 | 如果太长，观战会无聊 |
| 单位行为差异 | 明显 | 如果太相似，策略深度不足 |

### 代码验证

原型代码包含：
- `BattleManager` - 完整的回合制战斗逻辑
- `UnitVisual` - 单位视觉表示
- `test_battle.gd` - 无头测试脚本

## Metrics

### 代码复杂度
- 总行数: ~150 行 GDScript
- 类数量: 2 (BattleManager, UnitVisual)
- 依赖: 无外部依赖

### 性能预估
- 内存: < 1KB (纯数据)
- CPU: 最小 (简单数学运算)
- 帧时间: < 0.1ms/回合

## Recommendation: **PROCEED** ✅

原型验证了核心逻辑的可行性：

1. **回合制战斗**适合「观战」体验
2. **最近优先目标选择**足够简单且可预测
3. **单位差异**可以通过攻击范围和速度体现

### 如果继续开发，需要注意：

**架构要求**:
- 实现完整的 `UnitInstance` 类（包含 UnitDefinition 引用）
- 实现伤害计算系统（护甲、克制、随机性）
- 实现战斗状态机（Setup → Fighting → Victory/Defeat）

**性能目标**:
- 60 FPS 战斗动画
- < 100ms 回合处理时间
- 支持 20+ 单位同屏

**范围调整**:
- MVP: 保持 3x3 网格
- 后期: 可扩展到 4x4 或更大

**预估开发工作量**:
- 战斗系统核心: 2-3 天
- UI 和动画: 2-3 天
- 平衡调整: 1-2 天

## Lessons Learned

1. **回合制 vs 实时制**: 回合制更容易实现和调试，适合第一个游戏
2. **网格大小**: 3x3 是最小可行大小，策略空间足够但不大
3. **目标选择**: 最近优先是最简单的策略，可预测性高
4. **观战体验**: 需要动画和音效支持，否则纯数据展示会很无聊

## Next Steps

2. 调整单位属性，确保合理的胜率和战斗时长
3. 添加简单的 UI（HP 条、伤害数字）
4. 如果验证成功，开始生产实现
